Causality and Wiring Validation
1. Strict Slicing Entry Point: In [features.py](file:///c:/Users/ok/Desktop/Plivo/starter/features.py), the input audio array is clipped to the causal boundary at the very start of `extract_features_causal`: `x_causal = x[:int(pause_start * sr)]`. All helper functions and estimators have no reference to the original variable or its outer length `len(x)`.
2. Total Duration Leakage Prevention: No feature normalizes the timestamp or pause index against the total file duration, as that would communicate the "future" turn end. Per-pause statistics (pitch/energy medians) are computed strictly on the local context segment of `x_causal`.
3. Programmatic Causality Regression Test: We built `verify_causality_invariance` to copy an audio sample, corrupt all post-pause values with random noise, extract features, and assert that the feature vectors are bit-identical. This wiring guard passed successfully during training.
Metric Deep-Dive: Mean Delay vs Operating Point Delay
The Metric Distinction: You might notice that for English, the operating point delay is set to **450 ms**, but the reported OOF response delay is **1163 ms**. This is not a contradiction.
Mean Latency Composition: The *mean response delay* is calculated across all EOT turns in the folder. For turns where the model predicts EOT early (probability ≥ threshold), the delay is the operating point delay (450 ms). For turns where the model does not trigger, the system waits for the full fallback timeout (**1600 ms**). The average of these latencies yields 1163 ms, representing an honest baseline improvement over the starter's 1305 ms.
Sanity Check (Hindi vs. English): Hindi true EOT and hold pauses are systematically much shorter than English pauses (as verified in labels.csv). This allows the sweep on Hindi to operate at a very aggressive operating point (threshold=0.05, delay=850ms) to score 850 ms OOF delay, while English requires a higher threshold (0.45) to avoid cutoffs, resulting in a higher OOF delay.
