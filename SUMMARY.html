Correlation / Voting Ensemble Architecture (Cross-Language Generalization)
1. The "Unknown Language" Problem Solved: When the model runs on the hidden test set, it has no knowledge of whether the incoming audio is English or Hindi. Normalizing files together (via StandardScaler) would mash different language characteristics, creating cross-lingual scale warping. By removing standardizers entirely, we pass raw features directly to the model.
2. Uncorrelated Voting Trees: Our RandomForest classifier consists of 100 decision trees that act as an uncorrelated voting ensemble. Because different trees partition the space on different subsets of variables (e.g. Tree A splits on Pitch decay, Tree B on MFCC texture, Tree C on Breath spike), they must correlate and agree to output a high probability of EOT. This makes the model extremely robust, preventing it from being tripped by a single vocal anomaly or background noise.
3. Automatic Language Separation: Passing raw unscaled parameters allows the decision trees to naturally branch and separate stress-timed English cues from syllable-timed Hindi cues on their own, removing the need for a file prefix or language tag.
Causality and Wiring Validation
1. Strict Slicing Entry Point: In [features.py](file:///c:/Users/ok/Desktop/Plivo/starter/features.py), the input audio array is clipped to the causal boundary at the very start of `extract_features_causal`: `x_causal = x[:int(pause_start * sr)]`. All helper functions and estimators have no reference to the original variable or its outer length `len(x)`.
2. Total Duration Leakage Prevention: No feature normalizes the timestamp or pause index against the total file duration, as that would communicate the "future" turn end. Per-pause statistics (pitch/energy medians) are computed strictly on the local context segment of `x_causal`.
3. Programmatic Causality Regression Test: We built `verify_causality_invariance` to copy an audio sample, corrupt all post-pause values with random noise, extract features, and assert that the feature vectors are bit-identical. This wiring guard passed successfully during training.
Metric Deep-Dive: Mean Delay vs Operating Point Delay
The Metric Distinction: You might notice that for English, the operating point delay is set to **450 ms**, but the reported OOF response delay is **1137 ms**. This is not a contradiction.
Mean Latency Composition: The *mean response delay* is calculated across all EOT turns in the folder. For turns where the model predicts EOT early (probability ≥ threshold), the delay is the operating point delay (450 ms). For turns where the model does not trigger, the system waits for the full fallback timeout (**1600 ms**). The average of these latencies yields 1137 ms, representing an honest baseline improvement over the starter's 1305 ms.
Sanity Check (Hindi vs. English): Hindi true EOT and hold pauses are systematically much shorter than English pauses (as verified in labels.csv). This allows the sweep on Hindi to operate at a very aggressive operating point (threshold=0.05, delay=850ms) to score 850 ms OOF delay, while English requires a higher threshold (0.45) to avoid cutoffs, resulting in a higher OOF delay.

Testing Tools & Generalization
In-Sample Sanity Check (Not Generalization): Running [predict.py](file:///c:/Users/ok/Desktop/Plivo/predict.py) using the final fitted model on its own training directories yields response delays of 100 ms (English) and 100 ms (Hindi). Because this model was trained on this same data, these scores are optimistic train-fit numbers that serve strictly as an end-to-end pipeline sanity check. The only honest, reportable generalization performance metrics on unseen data are the OOF CV scores (1137 ms / 850 ms).
Path Generality Verification: The script [predict.py](file:///c:/Users/ok/Desktop/Plivo/predict.py) was tested against a mock directory with arbitrary folder names. The test verified that the file reading and output formatting are path-independent and generalize to unseen directories.
Optional Speaking Check: We developed [speak_test.py](file:///c:/Users/ok/Desktop/Plivo/speak_test.py) as an optional diagnostic tool. You can record a short 16 kHz mono WAV file, specify where you paused, and run `python speak_test.py --wav --pause_start ` to query the model probability.
